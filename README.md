## How do you write programs that are maintainable, readable, and adaptable?
The most significant choice I made for maintainability was to develop the CRUD module (AnimalShelter class) in Project One as an entirely isolated abstraction layer between the app and the database. Rather than throwing raw PyMongo calls all over the dashboard notebook, I only allow the database to be accessed through four methods (create, read, update, and delete), which receive and return only plain Python types. Therefore, Project Two's dashboard code can read the database via db.read({"breed": "Labrador"}) and handle the returned data without worrying about connection strings, authentication, or query syntax.
The real benefit of such a design was evident immediately in Project Two: in case I wanted to incorporate multiple compound filters (breed + sex + age range), I only needed to alter the dictionary I passed to db.read(), no changes at all to the module. Besides, the separation also significantly increases the readability of the code because a developer in the future who is checking the dashboard callback will easily grasp the data flow without having to decipher database driver syntax.
Looking ahead, this CRUD module can be reused or extended in a number of directions: for instance, a REST API layer built with Flask or FastAPI could call the same methods to expose the shelter data to mobile apps; a scheduled reporting script could use read() with date, range filters to generate weekly intake summaries; or the module could be adapted for a different collection (e.g., staff records, training outcomes) by only changing the database and collection names that are passed to the constructor.
## How do you approach a problem as a computer scientist?
I focused on three separate layers: the data model (imploring what MongoDB should store and how it should be queried), the business logic (which rules of filters would define the different rescue categories), and the presentation layer (what the user should see and be able to work with). Completing each layer, locally made, each one separately testable, I was able to check that the CRUD module gave the right records without any UI, and I could check if the layout was correct before even connecting the callbacks.
This was quite different from when we had problems in other classes that were self-contained, and the solution path was more or less prescribed. Here, the specs came from a simulated real client who had very specific operational needs (rescue, suitability criteria, a branded interface, geographic awareness), so I had to make design judgment calls instead of just implementing a known algorithm.

The iterative debugging process, in which each of the three runtime errors I encountered revealed deeper insights into how Dash's reactive system works, also felt very much like professional software development rather than academic problem-solving.
Next time I work on a database project for client needs, I will follow the same layered approach: starting by modeling the data around the key queries the client requires (not around each of the three runtime errors I encountered, which provided deeper insights into the data), then develop and test the data access layer separately, and finally consider the UI as a consumer of a stable API rather than tightly linking it to database logic.
## What do computer scientists do, and why does it matter?
Computer scientists redefine the problems of daily life so they can be scaled, automated, and run continuously, resulting in capabilities beyond human performance. For example, in theory, the staff at Grazioso Salvare could manually sift through thousands of shelter records to identify suitable candidates for rescue training. However, such a process would be very slow, prone to errors, and incapable of being carried out consistently by different staff members or in different shelter locations. My contribution was to map the criteria for rescue and suitability into the filtering logic, allowing the matching animals to be identified with a single click without experts having to come up with the answers each time.
In general, this sort of contribution helps to reduce the cost of good decisions. Thus, the time staff spends on data wrangling is reduced by clearly understanding at first glance the training requirements, matching animals, filtering for the rescue type in one click, and immediately getting a map showing the animal's location. The coming year will see continued efficiency gains; the same dashboard will support different intake cycles, be provided to partner shelters, and be supplemented with new filter criteria as training programs change. What makes software matter is not the technological stuff, but the human potential it unleashes.
